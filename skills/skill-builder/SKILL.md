# Skill Builder (Agent Skills MCP Server専用)

このスキルはAgent Skills MCP Server用の軽量で高品質なスキルを作成します。

## 目的

ユーザーの要求に基づいて、以下の要素を含む完全なSKILL.mdファイルを生成します：

- YAML frontmatter (name, description, metadata, allowed_tools)
- 目的セクション
- 使用方法セクション
- 具体的な実行例（2個以上）
- エラーハンドリング
- 注意事項

## 使用可能ツール

このMCPサーバーでは以下のツールが使用可能です：

1. **file_read**: ファイル読み込み（skills/, community-skills/, managed-skills/, .tmp/ のみ）
2. **file_write**: ファイル書き込み（managed-skills/, .tmp/ のみ）
3. **shell**: シェルコマンド実行（30秒タイムアウト）
4. **web_fetch**: Web取得（重要な制限あり）

### web_fetch の重要な制限

**取得可能**:
- 静的HTML（サーバー側でレンダリング済み）
- JSON APIエンドポイント
- XML/RSS フィード
- テキストデータ（100KB制限）

**取得不可**:
- JavaScript動的読み込みページ（React, Vue, SPAなど）
- ログインが必要なページ
- 100KB超のコンテンツ

**推奨アプローチ**:
1. **API優先**: 公式APIエンドポイントを探す（例: `/api/data.json`）
2. **静的HTML**: JavaScriptなしでも表示されるページを使用
3. **代替手段**: データ取得失敗時の対処法を必ず含める

## スキル作成の原則

### 0. **最重要**: スキルを汎用化する

スキルは特定の状況に特化せず、**パラメータ化**して再利用可能にすること：

**❌ 悪い例（特定状況に特化）**:
- スキル名: `marunouchi-lunch-finder`
- 説明: 「丸の内周辺のランチ情報を調べる」
- 問題: 他の場所では使えない

**✅ 良い例（汎用化・パラメータ化）**:
- スキル名: `restaurant-finder`
- 説明: 「指定した場所の指定した食事タイプ（朝食/ランチ/ディナー）のレストラン情報を調べる」
- 利点: 任意の場所・食事タイプで再利用可能

**汎用化の手順**:
1. ユーザーの要求から**可変要素**を抽出（場所、時間、カテゴリ、条件など）
2. それらを**パラメータ**として扱う
3. スキルの説明・実行例でパラメータの使い方を明示

**例**:
- 「丸の内のランチ」→ パラメータ: location="丸の内", meal_type="ランチ"
- 「今日の東京の天気」→ パラメータ: location="東京", date="今日"
- 「Python のチュートリアル」→ パラメータ: language="Python", content_type="チュートリアル"

### 1. 軽量性を重視

- 不要な説明は省略
- 簡潔で明確な指示のみ
- スクリプトファイルは作成しない（file_write で直接SKILL.md作成）

### 2. 実用性を重視

- 具体的な実行例を必ず含める（最低2個、異なるパラメータで）
- エラーケースの処理方法を記載
- ユーザーが理解しやすい言語で記述

### 2-1. **重要**: スキルは動的な情報取得手順を定義する

スキルは「情報そのもの」ではなく、「最新情報を取得する方法」を定義します：

**❌ 悪い例（静的な情報を含む）**:
```markdown
## 東京の天気予報

今日: 晴れ（最高気温: 15℃）
明日: 曇り（最高気温: 12℃）
```

**✅ 良い例（動的取得の手順を定義）**:
```markdown
## 使用方法

毎回実行時に WebFetch で最新データを取得します：

WebFetch(
  url="https://api.example.com/weather/tokyo",
  prompt="最新の天気予報データから今日と明日の天気を抽出"
)

実行のたびに最新の天気情報が取得されます。
```

**重要な原則**:
- 実行例の出力には「（例）」を明記し、実際には動的に変わることを示す
- 具体的な日付・数値は避け、プレースホルダー（「X月X日」「XX℃」）を使用
- 「最新」「現在」「リアルタイム」などの表現を使用

### 3. web_fetch 使用時のベストプラクティス

**CRITICAL: 具体的な実装を必ず含める**

web_fetch を使用するスキルでは、以下を**必ず**記載すること：

1. **具体的なAPIエンドポイントURL**
   - ❌ 悪い: "気象庁APIを使用"
   - ✅ 良い: `https://www.jma.go.jp/bosai/forecast/data/forecast/130000.json`

2. **実装手順を具体的に（WebFetch形式）**
   ```
   1. WebFetch(
        url="https://api.example.com/data",
        prompt="このJSONデータから{必要な情報}を抽出して整理してください"
      )
   2. 取得したデータをユーザー向けにフォーマット
   ```

3. **期待されるレスポンス例を含める**
   ```json
   {
     "data": {
       "temperature": 15,
       "weather": "晴れ"
     }
   }
   ```

**データソースの選択順**:
1. **公式API（確実に存在する場合のみ）**: JSON/XMLエンドポイント
   - **実在確認が必須**: URLが現在稼働中であることが確実な場合のみ記載
   - 実在するAPIのみを記載（架空のAPIは禁止）
   - 廃止されたサービスは絶対に記載しない
   - 不明な場合は次のステップへ

2. **公式サイトのURL案内（APIが不明な場合）**:
   - APIエンドポイントが不明な場合は、公式サイトのトップページURLのみ案内
   - 「最新情報は公式サイトでご確認ください」と明記
   - これが最も安全で確実なアプローチ

3. **静的HTML**: サーバー側レンダリングページ
   - JavaScriptなしで表示されることを確認
   - ただし、多くの現代サイトは動的なので注意

4. **エラーハンドリング**: データ取得失敗時の対処法を必ず含める
   - 公式サイトURLを案内（これは常に有効）

**例（天気情報）**:
- ✅ 良い: 気象庁API `https://www.jma.go.jp/bosai/forecast/data/forecast/{code}.json`
- ❌ 悪い: Yahoo天気（JavaScript動的読み込み）
- ❌ 悪い: "気象庁の花粉API"（実在しない架空のAPI）

**エラーハンドリング**:
- データ取得失敗時、ユーザーに公式URLを案内
- 簡潔なエラーメッセージ（長い技術的説明は不要）
- 代替データソースを提案（実在するサイトのみ）

### 4. フォーマット遵守

- 必要に応じて weather-forecast や example-skill を参考にする
- YAML frontmatter の構文を厳守
- Markdownの見出し構造を適切に使用

## スキル作成手順

1. **要件理解**: ユーザーの要求を分析
2. **参考確認**: 必要に応じて weather-forecast を file_read で確認
3. **データソース検証**（web_fetch使用時のみ）- **最重要**:
   - **実在確認が最優先**: URLが現在も稼働中であることを確認
   - **不明な場合は記載しない**: URLの正確性に自信がない場合は、APIエンドポイントを記載せず、公式サイトURLのみを案内
   - **廃止されたサービスに注意**: 古い情報（例: kafun.env.go.jp は廃止済み）を使用しない
   - 静的HTML/JSONであることを確認（JavaScript動的ページは不可）
   - 架空のAPIは絶対に記載しない

   **推奨アプローチ**:
   - ステップ1: 公式サイトのトップページURLを確認（これは確実）
   - ステップ2: APIドキュメントが見つかれば、そのエンドポイントを記載
   - ステップ3: APIが不明な場合は、「公式サイトで最新情報を確認」のみ記載
4. **SKILL.md生成**: file_write で managed-skills/{skill_name}/SKILL.md を作成
   - **重要**: web_fetch使用時は具体的なURLを「使用方法」セクションに明記
   - 実装手順を番号付きリストで具体的に記載
   - 期待されるレスポンス例を含める（JSON/HTML構造）
5. **完了報告**: 生成したスキルのパスと概要を報告

# Weather API Fetcher（汎用版）

気象庁の公式APIから**任意の地域**の天気予報データを取得します。

## 使用方法

### APIエンドポイント
```
https://www.jma.go.jp/bosai/forecast/data/forecast/{area_code}.json
```

### 主要都市コード
- 東京: 130000
- 大阪: 270000
- 名古屋: 230000
- 札幌: 016000
- 福岡: 400000

**パラメータ**: location（任意の日本の地域名）

## 実行例

### 例1: 東京の天気

**入力**: "東京の天気を教えて"

**パラメータ**: location="東京"

**処理**:
1. 地域名から地域コードを決定（東京 = 130000）
2. WebFetch を実行:
   ```
   WebFetch(
     url="https://www.jma.go.jp/bosai/forecast/data/forecast/130000.json",
     prompt="この気象庁の予報JSONデータから今日と明日の天気と気温を抽出して整理してください"
   )
   ```
3. 期待されるJSONレスポンス:
   ```json
   {
     "timeSeries": [{
       "timeDefines": ["2024-02-15T..."],
       "areas": [{
         "area": {"name": "東京地方"},
         "weathers": ["晴れ"],
         "temps": ["15"]
       }]
     }]
   }
   ```
4. WebFetch が自動的に `timeSeries[0].areas[0]` から天気と気温を抽出

**出力例**（実行時の最新データ）:
```
【東京の天気予報】（気象庁公式データ - 取得時刻: X月X日 XX:XX）
今日: 晴れ（最高気温: 15℃）
明日: 曇り（最高気温: 12℃）

※ 実行のたびに気象庁APIから最新データを取得します
```

### 例2: 大阪の天気（異なるパラメータ）

**入力**: "大阪の天気はどう？"

**パラメータ**: location="大阪"

**処理**:
1. 地域名から地域コードを決定（大阪 = 270000）
2. WebFetch を実行（URLのarea_codeのみ変更）

```

- **JSONパースエラー**: シンプルなエラーメッセージ

## エラーハンドリング

### よくある問題

0. **汎用化不足**（最も重要な問題）

   **問題**: 生成されたスキルが特定の状況にしか使えない

   **症状**:
   - スキル名に具体的な値が含まれる（例: `tokyo-weather`, `marunouchi-lunch`）
   - 説明文に具体的な固定値が含まれる（例: 「東京の天気を調べる」「丸の内のランチを探す」）
   - 実行例が1つのパターンしかない

   **原因**:
   - ユーザーの要求から可変要素を抽出していない
   - パラメータ化を忘れている

   **必須対策**:
   1. **可変要素を特定**: ユーザー要求から場所・時間・カテゴリなどの可変要素を抽出
   2. **パラメータ化**: それらをパラメータとして扱う
   3. **汎用的な命名**: スキル名と説明文は汎用的に
   4. **複数の実行例**: 異なるパラメータ値での実行例を2個以上含める

   **悪い例**:
   ```yaml
   name: marunouchi-lunch-finder
   description: 丸の内周辺のランチ情報を調べる
   ```

   **良い例**:
   ```yaml
   name: restaurant-finder
   description: 指定した場所の指定した食事タイプ（朝食/ランチ/ディナー）のレストラン情報を調べる
   ```

   **実行例の比較**:

   ❌ 悪い（パラメータなし）:
   ```markdown
   ## 実行例

   **入力**: "丸の内のランチを教えて"
   **出力**: （丸の内の結果）
   ```

   ✅ 良い（複数のパラメータセット）:
   ```markdown
   ## 実行例

   ### 例1: 東京駅周辺のランチ
   **入力**: "東京駅周辺のランチを教えて"
   **パラメータ**: location="東京駅周辺", meal_type="ランチ"

   ### 例2: 新宿のディナー
   **入力**: "新宿でディナーを探している"
   **パラメータ**: location="新宿", meal_type="ディナー"
   ```

1. **YAML構文エラー**
   - frontmatter の `---` が正しく閉じられているか確認
   - インデントがスペース2個で統一されているか確認

2. **metadata 型エラー**（最も多いエラー）
   - ❌ 間違い: `supported_languages: ["Japanese", "Tagalog"]` （配列形式）
   - ✅ 正解: `supported_languages: "Japanese, Tagalog"` （カンマ区切り文字列）
   - **重要**: metadata のすべての値は文字列である必要があります
   - 配列、数値、真偽値は使用できません

3. **ファイル書き込み失敗**
   - managed-skills/ ディレクトリが存在するか確認
   - パスが正しいか確認（managed-skills/{skill-name}/SKILL.md）

4. **web_fetch データ取得失敗**（最も重要）

   **問題**: 生成されたスキルが動作しない

   **原因**:
   - ❌ JavaScript動的読み込みページを指定（React, Vue, SPAなど）
   - ❌ 架空のAPIエンドポイントを記載（例: "気象庁の花粉API"は存在しない）
   - ❌ 廃止されたサービスのURLを記載（例: kafun.env.go.jp は既に廃止）
   - ❌ 具体的なURLを記載せず曖昧な説明のみ（例: "気象情報サイトから取得"）
   - ❌ URLの正確性を確認せずに記載

   **必須対策**:
   - ✅ 実在する公式APIエンドポイントURLを明記
   - ✅ 静的HTML/JSONのみを使用（JavaScript不要）
   - ✅ 具体的な実装手順を番号付きで記載
   - ✅ 期待されるレスポンス例を含める

   **良い例**:
   ```
   ### APIエンドポイント
   https://www.jma.go.jp/bosai/forecast/data/forecast/130000.json

   ### 実装手順
   1. WebFetch(
        url="https://www.jma.go.jp/bosai/forecast/data/forecast/130000.json",
        prompt="この気象庁の予報JSONデータから天気と気温を抽出してください"
      )
   2. WebFetch が自動的にJSONをパースして必要な情報を抽出
   ```

   **悪い例1（架空のAPI）**:
   ```
   気象庁の花粉APIから花粉情報を取得します。
   （具体的なURLなし、APIも実在しない）
   ```

   **悪い例2（廃止されたURL）**:
   ```
   WebFetch(
     url="https://kafun.env.go.jp/api/data",
     prompt="花粉情報を取得"
   )
   （このURLは既に廃止されており、アクセス不可）
   ```

   **悪い例3（不明なURL）**:
   ```
   tenki.jpのAPIから天気情報を取得します。
   （tenki.jpには公開APIが存在しない、JavaScript動的ページ）
   ```

   **推奨: APIが不明な場合**:
   ```
   ## エラーハンドリング

   現在、自動取得可能な公式APIが見つかりません。
   最新情報は以下の公式サイトでご確認ください：

   - 環境省 花粉観測システム: https://kafun.taiki.go.jp/
   - 日本気象協会: https://tenki.jp/pollen/

   ※ 利用可能なAPIが見つかり次第、このスキルを更新します
   ```

5. **生成されたスキルの品質が低い**
   - weather-forecast を file_read で参考に確認
   - 実行例が具体的で分かりやすいか確認
   - web_fetch使用時は必ずURLとレスポンス例を含める

## 注意事項

- **汎用化を最優先**（最重要）:
  - ❌ 避ける: 特定の値に限定したスキル（「東京の天気」「丸の内のランチ」）
  - ✅ 推奨: パラメータ化された汎用スキル（「指定した地域の天気」「指定した場所のレストラン」）
  - 必ず複数の実行例を含める（異なるパラメータ値で）
  - スキル名と説明文は汎用的に記述
- **scripts/ ディレクトリは作成しない**: すべてfile_write で直接SKILL.md作成
- **WebFetch 形式を使用**: `WebFetch(url="...", prompt="...")` の形式で記述（weather-forecast 参照）
- **URL の正確性を最優先**:
  - ✅ 確実に存在するURLのみを記載
  - ❌ 不明・未確認のURLは記載しない
  - ❌ 廃止されたサービス（kafun.env.go.jp等）は絶対に使用しない
  - 不明な場合は、公式サイトのトップページURLのみ案内
- **動的情報取得を優先**: スキルは「最新情報を取得する手順」を定義する
  - ❌ 避ける: 具体的な日付や数値をスキル内に記載
  - ✅ 推奨: WebFetch で毎回最新データを取得する手順を記載
- **実行例は「例」として明示**: 出力例には「※実行のたびに最新データ」などの注記を含める
- **エラーハンドリングは必須**: データ取得失敗時の公式サイトURL案内を必ず含める
- **web_fetch を積極的に使用**: ただしURLの正確性が確実な場合のみ
- **軽量性を保つ**: 不要な説明は省略し、実用的な内容のみ含める
- **参考例を活用**: weather-forecast を file_read で確認して同じスタイルを使用
- **適切な言語を選択**: ユーザーの要求言語に合わせて記述
